## 1. 架构概述

### 1.1 什么是热路径和冷路径

**热路径（Hot Path）**:
- 实时交易和决策
- 低延迟要求（< 100ms）
- 高可用性要求（99.9%+）
- 在线状态管理

**冷路径（Cold Path）**:
- 批处理和分析
- 延迟容忍（分钟级）
- 可用性要求较低
- 历史数据处理

### 1.2 为什么需要拆分

**问题**:
- Dagster 被赋予热路径职责
- 实时和批处理混在一起
- 性能和可用性要求冲突

**解决方案**:
- 明确职责边界
- 独立的技术栈
- 不同的 SLA

### 4.2 数据导出策略

**实时导出（热路径 → Parquet）**:
```python
class DataExporter:
    """数据导出器"""

    async def export_hourly(self):
        """每小时导出"""
        # 1. 导出最近1小时的订单
        await self._export_orders(hours=1)

        # 2. 导出最近1小时的成交
        await self._export_fills(hours=1)

        # 3. 导出最近1小时的库存变更
        await self._export_inventory_ledger(hours=1)

    async def export_daily(self):
        """每日导出"""
        # 1. 导出昨天的完整数据
        await self._export_orders(days=1)
        await self._export_fills(days=1)
        await self._export_positions(days=1)

        # 2. 压缩和归档
        await self._compress_and_archive()
```

**批量加载（Parquet → DuckDB）**:
```python
class DataLoader:
    """数据加载器（Dagster Asset）"""

    @asset
    def historical_orders(self) -> pd.DataFrame:
        """加载历史订单"""
        return duckdb.query("""
            SELECT * FROM read_parquet('data/orders/*.parquet')
        """).df()

    @asset
    def daily_pnl(self, historical_orders, historical_fills) -> pd.DataFrame:
        """计算每日盈亏"""
        # 使用 DuckDB 进行分析
        return duckdb.query("""
            SELECT
                date(timestamp) as date,
                market_id,
                SUM(pnl) as total_pnl
            FROM historical_fills
            GROUP BY date, market_id
        """).df()
```

## 6. 技术栈对比

### 6.1 热路径技术栈

| 层级 | 技术 | 原因 |
|------|------|------|
| 语言 | Python 3.11+ | 异步支持，生态丰富 |
| 数据库 | SQLite (WAL) / Postgres | 低延迟，事务支持 |
| 缓存 | 内存字典 / Redis | 极低延迟 |

| 监控 | Prometheus + Grafana | 实时指标 |
| 日志 | structlog | 结构化日志 |

### 6.2 冷路径技术栈

| 层级 | 技术 | 原因 |
|------|------|------|
| 编排 | Dagster | 数据管道编排 |
| 数据库 | DuckDB | OLAP 性能优秀 |
| 存储 | Parquet | 列式存储，压缩率高 |
| 分析 | Pandas + DuckDB | 数据分析 |
| 可视化 | Streamlit | 快速原型 |
| 调度 | Dagster Scheduler | 定时任务 |

## 8. 监控和告警

### 8.1 热路径监控

**关键指标**:
- 订单提交延迟 (p50, p95, p99)
- 行情更新延迟
- 库存查询延迟
- 订单成功率
- 系统可用性

**告警规则**:
- 延迟 > 100ms (警告)
- 延迟 > 500ms (严重)
- 订单失败率 > 5% (严重)
- 系统不可用 (紧急)

### 8.2 冷路径监控

**关键指标**:
- Dagster Job 成功率
- 数据导出延迟
- 回测完成时间
- Agent 评估准确率

**告警规则**:
- Job 失败 (警告)
- 数据导出延迟 > 1 小时 (警告)
- Agent 准确率 < 90% (警告)

## 9. P1-9 修复总结

### 9.1 明确的职责边界

✅ **热路径**:
- 市场数据、定价、路由、OMS、Watcher、Signer
- 实时、低延迟、高可用
- SQLite/Postgres

✅ **冷路径**:
- Dagster、DuckDB、Parquet、Agent 评估
- 批处理、分析、回测
- 可以容忍延迟

### 9.2 独立的技术栈

- 热路径：Python + SQLite/Postgres + asyncio
- 冷路径：Dagster + DuckDB + Parquet

### 9.3 清晰的数据流

- 热路径 → Parquet → 冷路径
- 定期导出，批量加载
- 职责分离，性能优化
