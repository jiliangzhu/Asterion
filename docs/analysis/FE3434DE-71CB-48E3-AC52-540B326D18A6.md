====================
二、你的评估必须覆盖这些问题
====================

A. 项目当前真实状态判断
请判断：
- 当前项目真正已经实现到了什么程度
- 当前项目是否真的达到了 “ready for controlled live rollout decision”
- 当前系统是否只是 demo，还是已具备较强基础设施价值
- 当前边界是什么
- 什么能力是已实现的，什么只是文档里提到但代码未真正落地的

B. 总体架构分析
请详细分析：
- 整体模块划分是否合理
- 当前仓库的分层是否清晰
- canonical contract 是否稳定
- persistence 分层是否合理：
  - trading.*
  - runtime.*
  - meta.*
  - agent.*
  - ui.*
- weather 域模型设计是否合理
- UI/read model 架构是否合理
- Dagster / cold path / scripts / tests 之间的协作是否清晰
- 是否存在平行 contract、重复表达、边界漂移的问题

C. 架构设计合理性评估
请评估当前架构设计是否合理，重点分析：
- 哪些设计是优秀的
- 哪些设计过度复杂
- 哪些设计虽然能跑，但长期维护风险高
- 哪些地方违反了“单一事实来源”“低耦合”“可回放”“可审计”等原则
- 哪些模块边界应该重构

D. 代码质量评估
请从工程质量角度审查：
- 代码结构
- 命名
- 模块边界
- 可读性
- 测试覆盖
- 可维护性
- 文档与代码一致性
- 是否存在“临时补丁越来越多、系统复杂度上升”的趋势

E. 安全 / 风险 / 漏洞分析
请重点关注：
- secrets / private key / env 配置风险
- controlled live boundary 是否真的足够安全
- 是否可能绕过审批或 env arm
- agent 是否可能越权影响执行链
- runtime 与 execution 之间是否有不安全耦合
- write queue / writerd / db path / file lock 是否存在生产风险
- replay/idempotency 是否足够稳健
- 远程 API 依赖是否存在单点故障
- UI 是否可能误导 operator
- 任何潜在的安全隐患、架构漏洞、运维风险、数据一致性风险

要求：
- 每个问题请给出风险等级：P0 / P1 / P2 / P3
- 尽量定位到具体文件和代码位置
- 对每个问题给出修复建议

F. 缺陷与技术债分析
请列出：
- 当前系统最明显的缺陷
- 哪些问题会阻碍未来扩展
- 哪些地方测试虽然通过，但实际生产风险仍高
- 哪些地方属于“能跑但不够稳”
- 哪些地方已经出现设计漂移或文档漂移

G. 交易与策略分析
请从真实代码能力出发，不要脱离当前实现空谈。

请评估：
- 当前天气市场策略链路是否合理
- 当前 market discovery / rule2spec / forecast / pricing / opportunity 的逻辑是否足够强
- 有没有明显的 alpha 流失点
- 哪些环节会导致错误定价
- 哪些环节会导致机会排序失真
- 当前策略是否真正有赚钱潜力
- 如果要提高盈利能力，最应该优化哪些环节

请重点分析：
- 市场筛选逻辑
- 机会排序逻辑
- fair value 计算逻辑
- 流动性/可执行性评估
- agent review 在交易链中的实际价值
- 多市场排序是否合理
- 是否应该引入更好的 opportunity score / actionability score / confidence score

H. 商业与变现能力评估
请判断当前项目更适合：
- 自营交易系统
- 信号/研究平台
- operator console / trading infra
- market intelligence / weather intelligence 产品
- AI review / agent-assisted trading ops 工具

并分析：
- 当前最现实的变现路径是什么
- 当前离稳定赚钱还差什么
- 哪些功能足以支持早期验证
- 哪些关键能力明显不足
- 哪些能力应该优先投资建设

I. 更优方案建议
请你站在更高水平的视角，提出更好的做法：
- 更好的架构方式
- 更好的 UI/read model 组织方式
- 更好的 opportunity ranking 方式
- 更好的策略优化方式
- 更好的 agent 使用边界
- 更好的 replay / readiness / controlled live 设计
- 更好的测试策略
- 更好的文档与工程治理方式

要求：
- 不要只说“可以优化”
- 必须说清楚“为什么更好”
- 必须说明落地方式
- 尽量给出分阶段实施建议
